# CAM16 v11 / Hellwig-Fairchild 2022 support plan

## Goal

Land support for the 2022 Hellwig/Fairchild revisions to CAM16 and CAM16-UCS while keeping the current CAM16/CAM16-UCS behavior available for callers and for compatibility with existing palettes, tests, and golden values.

Implementation direction: **copy-and-fork the CAM16 implementation rather than making one `cam16.dart` handle both formula sets**. The legacy implementation should remain boring and stable; v11 should live in new files/classes.

This document refers to:

- **legacy CAM16**: the formulas currently implemented in `lib/colorspaces/cam16.dart`, `cam16_viewing_conditions.dart`, and `hct_solver.dart`.
- **CAM16 v11**: the revised appearance-attribute formulas proposed in Hellwig & Fairchild, “Brightness, lightness, colorfulness, and chroma in CIECAM02 and CAM16” (2022).
- **UCS v11**: the revised uniform color spaces proposed in Hellwig & Fairchild, “Revising CAM16-UCS” (2022), fitted against CAM16 v11.

## Source-paper summary

### CAM16 v11 appearance-attribute changes

The 2022 paper keeps the CAT16 adaptation, opponent dimensions, hue angle, lightness `J`, surround factor `c`, chromatic induction factor `N_c`, and `z` structure, but changes the post-adaptation appearance attributes:

- `Nbb` and `Ncb` are no longer used in the revised achromatic/colorfulness equations.
- Achromatic white response becomes:
  - `Aw = 2 * R_aw + G_aw + 0.05 * B_aw - 0.305`
- Achromatic response for a stimulus becomes:
  - `A = 2 * R_a + G_a + 0.05 * B_a - 0.305`
- Lightness remains:
  - `J = 100 * (A / Aw)^(c * z)`
- Brightness becomes linear in lightness:
  - `Q = (2 / c) * (J / 100) * Aw`
- The legacy `t`-based path is removed for `C`, `M`, and `s`.
- A new hue eccentricity function is used:
  - `e_t = 0.0582 cos(h) - 0.0258 cos(2h) - 0.1347 cos(3h) + 0.0289 cos(4h) - 0.1475 sin(h) - 0.0308 sin(2h) + 0.0385 sin(3h) + 0.0096 sin(4h) + 1`
- Colorfulness, chroma, and saturation become linearized:
  - `M = kM * N_c * e_t * sqrt(a^2 + b^2)`
  - `C = 35 * M / Aw`
  - `s = 100 * M / Q`

Open verification item: the PDF contains a constant discrepancy for `kM` (`47` in Equation 23 and `43` in Appendix A / inverse summary). A plausible reading is that `47` belongs to the first simplified colorfulness formula derived from the old `t` numerator, while `43` is the final implementation constant after the subsequent refit/replacement of the hue eccentricity function. The phrase “derived from the numerator” appears to describe the surviving structure `N_c * e_t * sqrt(a^2 + b^2)`, not an algebraic derivation of the final scale constant from `(50000 / 13)`, because legacy `M` is obtained from `t` after division by an adapted-response denominator and multiple nonlinear rescalings. Treat `43` as the leading implementation candidate because Appendix A is the final summary and its inverse equations are internally consistent with `43`, but still verify against errata or an authoritative implementation before locking fixtures.

### UCS v11 changes

The 2022 UCS paper says the existing CAM16-UCS formulas must be refit because they depend on the legacy `M`, `C`, and `Q` behavior. It proposes two Euclidean color-difference spaces, both using CAM16 v11 attributes:

Relative UCS (`J'a'b'`):

- `J' = 1.7 * J / (1 + 0.007 * J)`
- `C' = 2.4 * ln(1 + 0.098 * C) / 0.098`
- `a' = C' * cos(h)`
- `b' = C' * sin(h)`

Absolute UCS (`Q'p't'`):

- `Q' = 0.86 * 1.7 * Q / (1 + 0.007 * J)`
- `M' = 2.0 * ln(1 + 0.094 * M) / 0.094`
- `p' = M' * cos(h)`
- `t' = M' * sin(h)`

UCS v11 uses ordinary Euclidean distance. Do **not** apply legacy CAM16-UCS distance compression `1.41 * dEPrime^0.63` to v11 distances.

In practical terms:

- Relative UCS (`J'a'b'`) compares colors relative to the scene/display white, so it is usually the better fit for palettes and object-color interpolation.
- Absolute UCS (`Q'p't'`) compares apparent brightness/colorfulness in absolute viewing conditions, so it is more useful when scene luminance/HDR/emissive brightness matters.

## Current code touch points

- `lib/colorspaces/cam16.dart`
  - Implements forward legacy CAM16 from ARGB/XYZ.
  - Stores legacy `jstar`, `astar`, `bstar` directly on `Cam16`.
  - `distance()` uses legacy compressed CAM16-UCS distance.
  - `xyzInViewingConditions()` implements the legacy inverse path.
- `lib/colorspaces/cam16_viewing_conditions.dart`
  - Computes cached legacy viewing-condition values.
- `lib/colorspaces/hct_solver.dart`
  - Inlines legacy inverse CAM16 heavily, including a fast sRGB path and generic gamut path.
  - The gamut boundary fallback is useful for both versions because hue is still derived from the same opponent dimensions.
- `lib/colorspaces/hct.dart`
  - Builds HCT from legacy CAM16 hue/chroma and L* tone.
  - Interpolation currently uses legacy CAM16-UCS coordinates.
- `lib/colorspaces/gamut.dart` and `matrix_utils.dart`
  - Gamut matrix support should stay usable. The CAT16/adaptation composite remains relevant to both versions.
- `lib/theming/palette.dart`
  - Uses `Hct`, `HctSolver.solveToInt()`, and `contrastingTone()` extensively.
  - Should accept a `ColorModel` parameter alongside `Algo`, then pass it through every HCT/solver call.
- `lib/theming/monet_theme_data.dart`
  - Owns the public theme-generation factories and already threads `Algo` through to `Palette`.
  - Should own a `ColorModel` field and thread it through the same way.
- `lib/theming/palette_lerped.dart`, `palette_snapshot.dart`, `animated_monet_theme.dart`, and `monet_theme.dart`
  - Need small pass-through/capture updates once `Palette` and `MonetThemeData` carry `ColorModel`.
- `lib/contrast/contrast.dart`
  - Uses HCT solving indirectly through `contrastingTone()` callers. Contrast algorithms stay independent of color appearance model.

## API design

### Color model enum

Use a package-level color-appearance-model enum rather than a CAM16-specific one. This keeps HCT/palette APIs from looking tied to one class name and leaves room for later models.

```dart
enum ColorModel {
  cam16,
  cam16v11,
}
```

Use `cam16v11` rather than `cam16_v11` so the enum value follows Dart lowerCamelCase style. Define this in a small public file, e.g. `lib/colorspaces/color_model.dart`, and export it from `lib/libmonet.dart`.

Default all existing public APIs to `ColorModel.cam16` at first. This preserves source behavior and test goldens. A later major release can consider changing defaults.

For UCS selection, keep a separate enum:

```dart
enum Cam16UcsSpace {
  legacyCompressed,
  legacyEuclidean,
  v11Relative,
  v11Absolute,
}
```

Existing legacy `Cam16.distance(other)` should remain unchanged. New model-aware distance helpers can accept `Cam16UcsSpace` explicitly.

### Copy-and-fork CAM16 classes

Do **not** make `Cam16` internally branch on a model enum. Instead:

- keep `lib/colorspaces/cam16.dart` and `Cam16` as legacy CAM16;
- keep `lib/colorspaces/cam16_viewing_conditions.dart` and `Cam16ViewingConditions` as legacy viewing conditions;
- add `lib/colorspaces/cam16_v11.dart` with `Cam16V11`;
- add `lib/colorspaces/cam16_v11_viewing_conditions.dart` with `Cam16V11ViewingConditions`.

This means:

- `Cam16` remains source-compatible and behavior-compatible.
- `Cam16V11` can have v11-specific semantics without overloaded field meanings.
- Existing `jstar`, `astar`, `bstar`, and `distance()` on `Cam16` keep meaning exactly what they mean today.
- V11 UCS coordinates can use clearer names/APIs instead of pretending legacy `jstar/astar/bstar` are the same thing.

Suggested v11 shape:

```dart
class Cam16V11 {
  final double hue;
  final double chroma;
  final double j;
  final double q;
  final double m;
  final double s;

  const Cam16V11(...);

  static Cam16V11 fromInt(int argb);
  static Cam16V11 fromIntInViewingConditions(
    int argb,
    Cam16V11ViewingConditions viewingConditions,
  );
  static Cam16V11 fromXyzInViewingConditions(
    double x,
    double y,
    double z,
    Cam16V11ViewingConditions viewingConditions,
  );

  Cam16UcsCoordinates relativeUcs(); // J'a'b'
  Cam16UcsCoordinates absoluteUcs(); // Q'p't'
  double distance(Cam16V11 other, {Cam16UcsSpace space = Cam16UcsSpace.v11Relative});
}
```

Do not support cross-model `Cam16.distance(Cam16V11)` directly. If a caller wants to compare models, they should choose an explicit conversion/space.

### Shared support code

Even with copy-and-fork, avoid duplicating tiny neutral utilities where it is harmless:

- matrix/vector helpers stay in `matrix_utils.dart`;
- gamut boundary machinery can stay shared where possible;
- constants that are truly identical can live in a private shared helper only if doing so does not make either class hard to read.

Bias toward duplicated clarity over abstract shared kernels. The goal is that someone can open `cam16_v11.dart` and see the whole v11 model.

### HCT and solver APIs

Do not silently change `Hct.from()`, `Hct.fromInt()`, `Hct.colorFrom()`, or current `HctSolver.solveToInt()` behavior. Add model-aware entry points that dispatch to either legacy or v11 implementation:

```dart
static int HctSolver.solveToIntForModel(
  double hueDegrees,
  double chroma,
  double lstar, {
  ColorModel model = ColorModel.cam16,
});

static (double, double, double) HctSolver.solveToLinrgbForModel(
  double hueDegrees,
  double chroma,
  double lstar, {
  Gamut? gamut,
  ColorModel model = ColorModel.cam16,
});
```

Implementation note: because current `solveToInt` has an optional positional `callSite`, avoid changing its signature in the first pass. Add explicit model-aware wrappers or v11-specific methods first, then consider cleanup later.

For `Hct`, prefer a model-aware wrapper at the HCT level, not inside `Cam16`:

```dart
Hct.from(hue, chroma, tone, {ColorModel model = ColorModel.cam16});
Hct.fromInt(argb, {ColorModel model = ColorModel.cam16});
Hct.fromColor(color, {ColorModel model = ColorModel.cam16});
Hct.colorFrom(hue, chroma, tone, {ColorModel model = ColorModel.cam16});
```

Internally, HCT chooses `Cam16` or `Cam16V11` based on `ColorModel`.

### Palette and theme APIs

Expose `ColorModel` in `Palette` and `MonetThemeData` similarly to the existing `Algo` parameter. Theming is the main high-level caller, so v11 should be selectable without dropping down to raw HCT APIs.

Suggested `Palette` changes:

```dart
class Palette {
  final Algo _algo;
  final ColorModel _colorModel;

  Palette.base({
    required Color baseColor,
    required Color baseBackground,
    double? backgroundTone,
    required double contrast,
    required Algo algo,
    required ColorModel colorModel,
  });

  factory Palette.from(
    Color color, {
    required double backgroundTone,
    double contrast = 0.5,
    Algo algo = Algo.apca,
    ColorModel colorModel = ColorModel.cam16,
  });

  factory Palette.fromColorAndBackground(
    Color color,
    Color background, {
    double contrast = 0.5,
    Algo algo = Algo.apca,
    ColorModel colorModel = ColorModel.cam16,
  });
}
```

Then update every palette HCT/materialization point to pass the model:

- `Hct.fromColor(_baseColor, model: _colorModel)`
- `Hct.fromColor(_baseBackground, model: _colorModel)`
- `Hct.from(..., model: _colorModel)`
- `Hct.colorFrom(..., model: _colorModel)`
- `HctSolver.solveToIntForModel(..., model: _colorModel)` or equivalent wrappers in border/contrast verification code

Include `_colorModel` in `Palette.operator ==` and `hashCode`.

Suggested `MonetThemeData` changes:

```dart
class MonetThemeData {
  final Algo algo;
  final ColorModel colorModel;

  MonetThemeData({
    ...
    this.algo = Algo.apca,
    this.colorModel = ColorModel.cam16,
  });

  MonetThemeData copyWith({
    Algo? algo,
    ColorModel? colorModel,
    ...
  });

  factory MonetThemeData.fromColor({
    ...
    Algo algo = Algo.apca,
    ColorModel colorModel = ColorModel.cam16,
  });
}
```

Thread `colorModel` through `fromColor`, `fromQuantizerResult`, `fromColors`, and each `Palette.from(...)` call. Include it in equality, hash/cache key, and `copyWith`.

Also update:

- `_createColorScheme(...)` to accept `ColorModel colorModel` and pass it to `Hct.fromColor`, `Hct.from`, `Hct.colorFrom`, and `Palette.from(Colors.red, ...)`.
- `MonetTheme` to expose `ColorModel get colorModel => monetThemeData.colorModel;`.
- `AnimatedMonetTheme` / `InterpolatedMonetThemeData` to pass through `begin.colorModel` when constructing intermediate `MonetThemeData`.
- `PaletteLerped` to lerp with the source palette's model, or require both palettes to have the same `colorModel` and assert/fallback if they differ.
- `PaletteSnapshot` to preserve/carry the captured palette's model in its `super.base(...)` call.

Default remains `ColorModel.cam16`. Palette/theming migration should be an explicit opt-in change, because colors will move.

## Implementation plan

### Phase 0: formula verification and fixtures

Before changing code:

- Verify the `kM` constant in the revised `M` equation (`43` vs `47`). Working hypothesis: `47` is an intermediate/body-text constant and `43` is the final implementation constant after the eccentricity refit.
- Verify whether the eccentricity formula expects `h` in radians. The paper writes `cos(h)` and derives `h = atan2(b, a)`, so radians are likely; tests should lock this down.
- Create a small independent reference script/notebook from the paper formulas for known colors and viewing conditions.
- Capture fixtures for black, white, red, green, blue, and midgray under default viewing conditions for both legacy and v11.

### Phase 1: copy the legacy files

Create the v11 files as direct copies first:

- `lib/colorspaces/cam16_v11.dart`
- `lib/colorspaces/cam16_v11_viewing_conditions.dart`

Then rename classes and imports only:

- `Cam16` -> `Cam16V11`
- `Cam16ViewingConditions` -> `Cam16V11ViewingConditions`

At the end of this phase, v11 classes may still behave like legacy CAM16, but tests should prove the copied code compiles and the legacy classes remain unchanged.

### Phase 2: apply v11 forward formulas in the fork

Modify only `cam16_v11.dart` and `cam16_v11_viewing_conditions.dart`:

- remove v11 dependence on `Nbb`/`Ncb` in the revised achromatic/colorfulness equations;
- compute v11 `Aw` as `2 * R_aw + G_aw + 0.05 * B_aw - 0.305`;
- compute stimulus `A` as `2 * R_a + G_a + 0.05 * B_a - 0.305`;
- preserve `J = 100 * (A / Aw)^(c * z)`;
- compute `Q = (2 / c) * (J / 100) * Aw`;
- replace legacy `t`-based `C`, `M`, and `s` with the v11 equations;
- add the v11 hue eccentricity function.

Keep legacy `cam16.dart` untouched except for imports/exports if needed.

### Phase 3: UCS support

Implement UCS as separate coordinate calculations:

- `legacyCompressed`: existing `Cam16.distance()` result;
- `legacyEuclidean`: existing `jstar/astar/bstar` Euclidean distance without compression, useful for diagnostics;
- `v11Relative`: `J'C'a'b'` formulas using v11 `J` and `C`;
- `v11Absolute`: `Q'M'p't'` formulas using v11 `Q` and `M`.

For HCT interpolation, keep the default legacy path. Add a new overload/variant for v11 interpolation and default that v11 path to relative UCS unless there is a strong reason to choose absolute UCS.

### Phase 4: inverse CAM16 v11

Implement the v11 inverse in the fork as a clear, generic path first; optimize later.

For a target `(J, C, h)`:

1. Compute `Q` from revised `Q` formula if needed.
2. Convert `C` to `M`: `M = C * Aw / 35`.
3. Compute `e_t(h)`.
4. Convert `M` to opponent radius: `sqrt(a^2 + b^2) = M / (kM * N_c * e_t)` after `kM` is verified.
5. Set opponent `a = radius * cos(h)`, `b = radius * sin(h)`.
6. Compute `A = Aw * (J / 100)^(1 / (c * z))` and `p2 = A + 0.305`.
7. Reuse the linear solve from `(p2, a, b)` to adapted cone responses, then inverse chromatic adaptation, undiscouting, XYZ, and RGB.

Use the existing gamut-boundary bisection structure for out-of-gamut requests; hue still comes from the same opponent coordinates.

Only after correctness is locked down, add an sRGB-specialized v11 fast path analogous to `_findResultByJSrgb`.

### Phase 5: HCT and theming opt-in

Add opt-in v11 HCT creation and solving:

```dart
Hct.from(hue, chroma, tone, model: ColorModel.cam16v11);
Hct.fromInt(argb, model: ColorModel.cam16v11);
Hct.fromColor(color, model: ColorModel.cam16v11);
Hct.colorFrom(hue, chroma, tone, model: ColorModel.cam16v11);
```

Then expose the same opt-in at the theming layer:

```dart
Palette.from(color, backgroundTone: tone, colorModel: ColorModel.cam16v11);
MonetThemeData.fromColor(..., colorModel: ColorModel.cam16v11);
MonetThemeData.fromColors(..., colorModel: ColorModel.cam16v11);
```

Default remains `ColorModel.cam16`. Palette/theming adoption should be an explicit opt-in because colors will move.

### Phase 6: migration policy

Recommended rollout:

1. Release with legacy default and v11 opt-in.
2. Document differences and provide examples.
3. Add a package-level migration flag only if a major release plans to flip defaults.
4. If defaults ever change, keep legacy available indefinitely through explicit `model: ColorModel.cam16` and explicit legacy UCS selection.

## Testing plan

Add tests in `test/colorspaces`:

- Legacy regression tests: current `hct_test.dart` CAM16 values should pass unchanged.
- V11 forward fixtures: black, white, red, green, blue, and midgray under default v11 viewing conditions.
- V11 inverse round trips:
  - `Cam16V11.fromInt(argb).viewed(Cam16V11ViewingConditions.standard)` returns original or close ARGB.
  - `Hct.from(..., model: ColorModel.cam16v11)` preserves hue/tone and does not exceed requested chroma except at gamut boundaries.
- UCS distance tests:
  - legacy default `distance()` remains unchanged.
  - v11 distances are Euclidean and not compressed.
  - v11 relative coordinates are invariant to scene luminance where expected; v11 absolute coordinates scale with scene luminance where expected.
- Gamut tests:
  - `solveToLinrgbForModel(..., model: ColorModel.cam16v11, gamut: Gamut.displayP3)` remains bounded and preserves tone.
- Theming tests:
  - `Palette.from(...)` and `MonetThemeData.fromColor(...)` default to `ColorModel.cam16` and reproduce existing goldens.
  - Passing `colorModel: ColorModel.cam16v11` reaches all HCT/solver materialization points.
  - Equality/hash/cache keys distinguish otherwise-identical themes with different `colorModel` values.

Run only targeted tests, e.g. `flutter test test/colorspaces/hct_test.dart`, `flutter test test/colorspaces/gamut_test.dart`, or a specific theming test file.

## Documentation

Update API docs to explain:

- `ColorModel.cam16` is retained for compatibility and parity with Material Color Utilities-derived behavior.
- `ColorModel.cam16v11` is the scientifically recommended revised appearance model.
- UCS v11 should be used for color differences only, not as perceptual attributes.
- relative UCS (`J'a'b'`) vs absolute UCS (`Q'p't'`) selection.
- default behavior and how to opt into either version explicitly.

## Risks

- **Formula ambiguity**: the parsed paper text has at least one suspicious constant discrepancy. Do not implement until resolved.
- **Golden churn**: palette, contrast, and HCT outputs will change under v11. Keep opt-in until a deliberate migration.
- **Performance regression**: the current solver is heavily optimized. Land v11 generic first, benchmark, then specialize.
- **API confusion**: avoid names like `new`/`old`; use `cam16` and `cam16v11`/paper-year language.
- **Duplication drift**: copy-and-fork makes formulas easier to audit, but fixes to shared mechanics may need to be applied twice. Keep tests around both implementations.

## Open questions

1. What is the authoritative `kM` constant in the revised `M` formula?
2. Should v11 become the default in the next major version, or remain opt-in indefinitely?
3. Which UCS space should v11 HCT interpolation use by default: relative UCS for palette/object-color behavior, absolute UCS for scene-brightness behavior, or an explicit parameter?
4. If `PaletteLerped` receives palettes with different `ColorModel` values, should it assert, use the begin model, use the end model, or snapshot both sides first?
